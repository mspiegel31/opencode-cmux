# v1.2 Milestone Plan

**Created:** 2026-03-20 (supersedes earlier sketch of same date)
**Depends on:** v1.0 (shipped), v1.1 (resolved)
**Target:** npm `@mspiegel31/opencode-cmux@1.2.0`

---

## Summary

v1.2 extends the plugin's scope in three directions:

1. **Config foundation** ŌĆö replace the hand-rolled partial-merge with a Zod schema,
   switch config parsing to `jsonc-parser` (comment-aware), bootstrap the default
   config file with JSONC comments, and embed a `$schema` field pointing at the
   hosted JSON Schema URL.

2. **Event coverage expansion** ŌĆö add sidebar status management as a continuous
   state machine (working / waiting / done / error), extend desktop notifications
   to cover permission requests, add per-event `notify.*` flags to config, and
   research + wire up question-event detection.

3. **Schema hosting** ŌĆö generate the JSON Schema from the Zod schema at CI time
   and publish it via GitHub Pages on every push to `main`. No commit-back to the
   repo. Stable URL: `https://mspiegel31.github.io/opencode-cmux/schema.json`.

**Phase dependency:** Phase 1 must complete before Phase 3 (the schema generation
script imports the Zod config schema). Phase 2 is parallel-safe with Phase 3.

---

## Background: decisions made during milestone planning

| Question | Decision |
|---|---|
| Pages trigger | Push to `main` on every merge (simpler, always in sync with latest config shape) |
| Config file extension | Keep `.json` ŌĆö `jsonc-parser` handles it transparently; renaming breaks existing installs |
| Bootstrapped default config | Include JSONC comments explaining each field ŌĆö primary UX win of switching parsers |
| Sidebar vs. notify config | `sidebar.enabled` global toggle gates all sidebar calls; `notify.*` per-event flags gate desktop notifies independently |
| Permission request output | Both desktop notify AND persistent sidebar status (red lock icon, clears on `permission.replied`) |
| Question event detection | Resolved: `question.asked` / `question.replied` / `question.rejected` event types exist in the opencode event stream (confirmed via Caso's plugin full source). Not yet in our `EventType` enum ŌĆö needs adding. |

## Target config shape (v1.2)

```jsonc
{
  // JSON Schema for editor validation and autocomplete
  "$schema": "https://mspiegel31.github.io/opencode-cmux/schema.json",

  // Desktop notification settings
  "notify": {
    "sessionDone": true,      // notify when primary session goes idle
    "sessionError": true,     // notify when primary session errors
    "permissionRequest": true,// notify when OpenCode is blocked waiting for permission
    "question": true          // notify when agent asks the user a question
  },

  // Sidebar status bar settings (cmux set-status / clear-status)
  "sidebar": {
    "enabled": true           // master switch; set false if you have no sidebar space
  },

  // Subagent TUI pane viewer
  "cmuxSubagentViewer": {
    "enabled": true
  }
}
```

**Migration compatibility:** The old config shape (`cmuxNotify.enabled`,
`cmuxSubagentViewer.enabled`) must continue to work. The Zod schema uses
`.optional()` and `.default()` throughout; unknown top-level keys are stripped
(not errored) via `.strip()`.

Old keys that map cleanly:
- `cmuxNotify.enabled: false` ŌåÆ sets all `notify.*` flags to `false`

Old keys with no direct mapping (e.g. `cmuxNotify.enabled: true`) ŌåÆ ignored; new
defaults apply. This is a non-breaking change for the common case (all defaults
are `true`).

## Phase 1 ŌĆö Config Foundation

**Blocking:** Phases 2 (partially) and 3 (fully) depend on this phase completing.

### 1-01: Zod schema + jsonc-parser migration

**Files touched:** `src/config.ts`, `package.json`

**Tasks:**

1. Add `zod` and `jsonc-parser` to `dependencies` in `package.json`.

2. Rewrite `src/config.ts`:
   - Define the config schema as a `z.object(...)` with nested objects, using
     `.default()` at every level so partial user configs are filled in without
     manual spreading.
   - Export `ConfigSchema` (the Zod schema object) and `Config` (the inferred
     type via `z.infer<typeof ConfigSchema>`).
   - Replace `JSON.parse(raw)` with `jsonc-parser`'s `parse()` so config files
     may contain `//` and `/* */` comments.
   - Call `ConfigSchema.parse(parsed)` for runtime validation. On `ZodError`:
     log a structured warning (path + message) via `console.error`, then fall
     back to `ConfigSchema.parse({})` (all defaults).
   - Remove the hand-rolled `{ ...DEFAULT_CONFIG.cmuxNotify, ...parsed.cmuxNotify }`
     spread pattern entirely.

3. Bootstrap: when the config file does not exist, write it with `jsonc-parser`'s
   `applyEdits` or a hand-written JSONC template string containing comments that
   explain each field. Include the `$schema` field at the top.

4. Env var overrides: migrate `CMUX_NOTIFY_ENABLED` and
   `CMUX_SUBAGENT_VIEWER_ENABLED` to map to the new config shape. Keep both
   working for backward compatibility.

**Schema structure (Zod):**

```ts
const NotifySchema = z.object({
  sessionDone:        z.boolean().default(true),
  sessionError:       z.boolean().default(true),
  permissionRequest:  z.boolean().default(true),
}).default({})

const SidebarSchema = z.object({
  enabled: z.boolean().default(true),
}).default({})

const SubagentViewerSchema = z.object({
  enabled: z.boolean().default(true),
}).default({})

export const ConfigSchema = z.object({
  $schema:            z.string().optional(),
  notify:             NotifySchema,
  sidebar:            SidebarSchema,
  cmuxSubagentViewer: SubagentViewerSchema,
}).strip()
```

**Acceptance criteria:**
- `ConfigSchema.parse({})` returns all defaults (no file required).
- `ConfigSchema.parse({ notify: { sessionDone: false } })` returns
  `{ notify: { sessionDone: false, sessionError: true, permissionRequest: true }, sidebar: { enabled: true }, ... }`.
- JSONC file with `// comments` parses without error.
- Malformed JSON logs a warning and returns defaults (does not throw).
- Old config shape `{ "cmuxNotify": { "enabled": false } }` does not cause
  an unhandled error (unknown keys are stripped, defaults apply).
- All existing `config.test.ts` tests pass (update assertions for new shape).
- New tests cover: JSONC comment passthrough, ZodError fallback, old shape
  no-crash.

## Phase 2 ŌĆö Event Coverage Expansion

**Parallel-safe with Phase 3.** Can begin once Phase 1 is complete.

### 2-01: Sidebar status state machine

**Files touched:** `src/cmux-subagent-viewer.ts`, `src/lib/cmux-utils.ts`

**Context:** Subagent panes already call `cmux set-status` / `cmux clear-status`
for individual agents (`agent-0`, `agent-1`, ŌĆ”). This task adds a parallel status
machine for the *primary* session ŌĆö a single status key `opencode` that reflects
what the primary session is doing right now.

**Status key:** `opencode` (distinct from per-agent `agent-N` keys).

**State transitions:**

| Event | Condition | Sidebar state | Icon | Color |
|---|---|---|---|---|
| `session.status` (`status.type === "busy"`) | `!isWaitingForInput()` | `Working` | `terminal` | `#F59E0B` (amber) |
| `session.status` (`status.type === "idle"`) | `!isWaitingForInput()`, primary session | clear status | ŌĆö | ŌĆö |
| `session.error` | primary session | clear status | ŌĆö | ŌĆö |
| `permission.asked` or `permission.updated` | new requestID | `Waiting` | `lock` | `#EF4444` (red) |
| `permission.replied` | `!isWaitingForInput()` after delete | clear or revert to Working | ŌĆö | ŌĆö |
| `question.asked` | new questionID | `Question` | `help-circle` | `#A855F7` (purple) |
| `question.replied` / `question.rejected` | `!isWaitingForInput()` after delete | clear or revert to Working | ŌĆö | ŌĆö |

**Note on `session.status`:** This event fires with `status.type: "busy" | "idle" | "retry"`,
and is more granular than `session.idle`. Caso's plugin uses this exclusively for
the working/done transitions ŌĆö our plugin currently uses `session.idle` only. The
sidebar machine will switch to `session.status` for fidelity; `session.idle` can
remain as a fallback for desktop notify.

**Config gate:** all sidebar calls check `config.sidebar.enabled` before firing.
If `false`, the state machine is disabled entirely (no `set-status` calls).

**Implementation notes:**
- The state machine lives in `CmuxPlugin` (`cmux-notify.ts`), not in
  `SubagentPaneManager` ŌĆö it's about the primary session, not subagent panes.
- Track a `sessionBusy` boolean to distinguish "permission arrived while working"
  from "permission arrived while idle" ŌĆö so `permission.replied` can revert to
  `Working` rather than clearing if the session is still active.
- Use a shared helper `setSidebarStatus(key, label, icon, color)` and
  `clearSidebarStatus(key)` in `src/lib/cmux-utils.ts` to avoid repeating CLI
  invocations in both plugins.

**Acceptance criteria:**
- `cmux set-status opencode Working ...` fires when assistant turn begins.
- `cmux set-status opencode Waiting --icon lock --color #EF4444` fires on
  `permission.updated`.
- `cmux clear-status opencode` fires on `session.idle` and `session.error`.
- On `permission.replied`: clears to `Working` if session was busy, else clears.
- `sidebar.enabled: false` suppresses all `set-status` / `clear-status` calls.
- No sidebar calls fire for subagent sessions (those use `agent-N` keys).

### 2-02: Permission request desktop notification

**Files touched:** `src/cmux-notify.ts`

**Context:** When OpenCode is blocked waiting for the user to approve a tool call,
the current plugin is silent. This is the highest-priority missing notification
since a missed permission request leaves the session hung indefinitely.

**Event surface (confirmed from Caso's source):**
- `permission.asked` ŌĆö v2 event name, fires when permission is pending.
- `permission.updated` ŌĆö v1 fallback name; handle both with `||`.
- `permission.ask` hook ŌĆö fires synchronously *before* the event, earlier in the
  permission pipeline. Caso uses this to register the pending ID eagerly so that
  `session.status idle` (which fires around the same time) doesn't prematurely
  clear the sidebar. **Use this hook for eager registration; use the event for the
  notify.**
- Deduplication: track seen `requestID`s in a `Set<string>`; the same permission
  can fire multiple events.

**Tasks:**

1. In `CmuxPlugin`'s event handler, handle `permission.updated`:
   - Guard: only fire if `config.notify.permissionRequest` is `true`.
   - Deduplicate by `requestID` (track seen IDs in a `Set`).
   - Fire: `cmux notify --title "OpenCode" --subtitle "Permission Required" --body "<tool name>"`.
   - On `permission.replied`: remove the `requestID` from the seen set.

2. The sidebar status for the same event is handled in Phase 2-01. This task
   is only the desktop notify.

**Acceptance criteria:**
- Desktop notify fires once per unique permission request.
- Does not fire for subagent sessions (guard with `isSubagent()` check).
- `notify.permissionRequest: false` suppresses the notify.
- Deduplication: two rapid `permission.updated` events with the same `requestID`
  produce exactly one notify.

### 2-03: Question event ŌĆö implement

**Status:** Ō£ō Research complete. Event types confirmed via Caso's plugin full source.

**Background:** The `ask` tool pauses the agent to request user input. Caso's
plugin sets a sidebar status ("question", purple) and fires a desktop notify when
this happens. The opencode event stream emits dedicated `question.*` event types ŌĆö
confirmed by reading `0xCaso/opencode-cmux` `src/index.ts` in full (uncompressed).

**Confirmed event types:**

| Event type | When it fires | Key payload |
|---|---|---|
| `question.asked` | Agent uses the `ask` tool, question is pending | `properties.questions[0].header` ŌĆö the question text |
| `question.replied` | User answered the question | `properties.id` or `properties.requestID` |
| `question.rejected` | Question was dismissed/cancelled | same as replied |

These are not in our `EventType` enum yet ŌĆö they exist in the event stream but
are cast via `as any` in Caso's plugin. Confirmed that `@opencode-ai/plugin` SDK
does not declare them in its `.d.ts` types (we checked `index.d.ts` directly).

**Additional findings from Caso's source:**

- `session.status` fires with `status.type === "busy"` / `"idle"` ŌĆö more granular
  than our current `session.idle` event. We're missing the `busy` branch (needed
  for 2-01 "working" state).
- `permission.asked` is a v2 event name; `permission.updated` is the v1 fallback.
  Caso handles both: `e.type === "permission.asked" || e.type === "permission.updated"`.
- The pending-state approach: `pendingPermissions` and `pendingQuestions` as
  `Set<string>` with `isWaitingForInput()` gating the `idle ŌåÆ done` path. This
  prevents the sidebar being prematurely cleared when the session goes "idle"
  (model stops generating) while still blocked on user input.

**Tasks:**

1. Add to `EventType` enum in `src/lib/plugin-base.ts`:
   ```ts
   QuestionAsked    = "question.asked",
   QuestionReplied  = "question.replied",
   QuestionRejected = "question.rejected",
   SessionStatus    = "session.status",    // if not already present
   PermissionAsked  = "permission.asked",  // v2 alias alongside existing PermissionUpdated
   ```

2. In `CmuxPlugin` (`src/cmux-notify.ts`), implement the full state machine
   matching Caso's approach:
   - Track `pendingPermissions: Set<string>` and `pendingQuestions: Set<string>`.
   - `isWaitingForInput()` returns true if either set is non-empty.
   - On `question.asked`: add ID to `pendingQuestions`, set sidebar status
     "question" (purple, `help-circle` icon), fire desktop notify if
     `config.notify.question === true`. Payload: `e.properties.questions?.[0]?.header`.
   - On `question.replied` / `question.rejected`: delete ID from `pendingQuestions`.
     If `!isWaitingForInput()`, restore sidebar to "working" (if session still busy)
     or clear.
   - Gate session-idle handling: if `isWaitingForInput()`, skip the done notify
     and status clear (session went idle because model stopped, not because it
     finished).

3. Add `notify.question` to `NotifySchema` in `src/config.ts` (can be done in
   Phase 1 since we know the field name now).

**Config addition (add to Phase 1-01 schema):**
```ts
const NotifySchema = z.object({
  sessionDone:        z.boolean().default(true),
  sessionError:       z.boolean().default(true),
  permissionRequest:  z.boolean().default(true),
  question:           z.boolean().default(true),  // ŌåÉ add this
}).default({})
```

**Acceptance criteria:**
- `question.asked` triggers sidebar status "question" (purple) and desktop notify.
- `question.replied` / `question.rejected` clears the question state.
- `session.status idle` while a question is pending does NOT fire the done notify.
- `notify.question: false` suppresses the desktop notify (sidebar status still fires
  if `sidebar.enabled: true`).
- `sidebar.enabled: false` suppresses sidebar status changes regardless of
  `notify.question`.

import { zodToJsonSchema } from "zod-to-json-schema"

const schema = zodToJsonSchema(ConfigSchema, {

### 3-02: GitHub Pages deployment workflow

**Files touched:** `.github/workflows/pages.yml`

**One-time manual prerequisite (not automated):**
> In the GitHub repo Settings ŌåÆ Pages ŌåÆ Source: set to "GitHub Actions".
> This step cannot be done by a workflow and must be done by the repo owner once.

**Workflow design:**

```yaml
name: Pages

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  deploy:
    name: Deploy schema to GitHub Pages
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build (required for schema import)
        run: npm run build

      - name: Generate schema
        run: npm run generate-schema

      - name: Prepare pages artifact
        run: |
          mkdir -p _site
          cp schema.json _site/schema.json

      - name: Upload pages artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: _site

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

**Design notes:**
- `npm run build` runs before `generate-schema` because the script imports
  `ConfigSchema` from the compiled output (`../src/config.js` resolves via
  `moduleResolution: bundler`). Alternatively the script can import directly from
  `src/config.ts` if `tsx` handles the TS import ŌĆö implementer should verify.
- No commit-back step. The schema lives only in the Pages deployment.
- `cancel-in-progress: false` on the concurrency group prevents a deploy from
  being cancelled mid-flight by a rapid second push.
- The `release.yml` and `pages.yml` both trigger on push to `main`; they are
  independent jobs and can run concurrently.

**Stable URL:** `https://mspiegel31.github.io/opencode-cmux/schema.json`

**Acceptance criteria:**
- Pushing to `main` triggers the workflow.
- `schema.json` is accessible at the stable URL after the workflow completes.
- The workflow does not commit any files back to the repo.
- The `pages.yml` workflow uses `actions/upload-pages-artifact@v3` and
  `actions/deploy-pages@v4` (not the legacy `peaceiris/actions-gh-pages`).
- The deployed `schema.json` is valid JSON Schema Draft 7 (or whichever draft
  `zod-to-json-schema` targets by default).

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Zod parse breaks existing user configs | Unknown keys stripped (not errored); ZodError falls back to defaults; migration test covers old `cmuxNotify` shape |
| `jsonc-parser` adds cold-start latency | It's the same lib opencode uses ŌĆö already in Node module cache when plugin loads |
| `permission.updated` fires multiple times per request | Deduplicate by `requestID` in a `Set` |
| `question.*` events not in SDK types | Cast via `as any` + string match, same as Caso's approach. Add to `EventType` enum for type safety. |
| Pages deploy races with release deploy | They are independent jobs; release only publishes npm, Pages only publishes schema.json ŌĆö no shared state |
| `generate-schema` imports TS directly vs compiled | Implementer must verify `tsx` resolution; fallback is import from `dist/` after build step |
| Adding `zod` increases install size | Zod v3 minified is ~13 KB gzipped ŌĆö acceptable for a plugin; zod v4 is even smaller if available |
